# getDiagnostics

Отримує діагностику мовного сервера (помилки, попередження, підказки) з підʼєднаної IDE для конкретного файлу або для кожного файлу, що IDE наразі тримає відкритим. Використовується для перевірки, що зміни коду компілюються без помилок перед оголошенням завдання завершеним.

## Коли використовувати

- Після `Edit` чи `Write` для підтвердження, що зміна не внесла помилку типу, синтаксичну помилку чи попередження лінтера.
- Перед завершенням завдання, щоб перевірити кожен відкритий файл на нерозвʼязані проблеми.
- При діагностиці помилки, про яку повідомляє користувач — витягування точного повідомлення компілятора чи перевірки типів з IDE уникає здогадок.
- Як легка альтернатива запуску повної збірки чи тестової команди, коли вам потрібно лише перевірити статичну коректність.

НЕ покладайтеся на `getDiagnostics` як на заміну тестового набору. Він повідомляє, що бачить мовний сервер, а не що виконується під час тестування чи в проді.

## Параметри

- `uri` (string, необовʼязковий): URI файлу (зазвичай `file:///absolute/path`), для якого отримати діагностику. Коли опущено, інструмент повертає діагностику для кожного файлу, що IDE наразі тримає відкритим.

## Приклади

### Приклад 1: Перевірка одного файлу після редагування

```
getDiagnostics(
  uri="file:///Users/sky/project/src/auth.ts"
)
```

Повертає будь-які помилки TypeScript, попередження ESLint чи інші повідомлення мовного сервера для `src/auth.ts`.

### Приклад 2: Огляд усіх відкритих файлів

```
getDiagnostics()
```

Повертає діагностику по кожному поточно відкритому редактору. Корисно наприкінці багатофайлового рефакторингу, щоб переконатися, що нічого не регресувало деінде.

## Примітки

- `getDiagnostics` — інструмент моста IDE. Він доступний лише коли Claude Code підʼєднаний до підтримуючої інтеграції IDE (наприклад, розширення VS Code). У простій термінальній сесії інструмент не зʼявиться.
- Результати відображають будь-які мовні сервери, які IDE завантажила — TypeScript, Pyright, ESLint, rust-analyzer тощо. Якість і покриття залежать від налаштувань IDE користувача, а не від Claude Code.
- Діагностика жива. Після редагування дайте мовному серверу мить для повторного аналізу перед інтерпретацією порожнього результату як успіху — перезапустіть, якщо файл щойно збережено.
- Рівні серйозності зазвичай включають `error`, `warning`, `information` та `hint`. Спочатку фокусуйтеся на `error`; попередження можуть бути навмисним стилем проєкту.
- Для файлів, що наразі не відкриті в IDE, мовний сервер може не мати діагностики для звіту, навіть якщо файл містить справжні проблеми. Відкрийте файл чи запустіть збірку для авторитетного покриття.
