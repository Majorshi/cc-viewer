# WebFetch

Забирает содержимое публичной веб-страницы, конвертирует HTML в Markdown и прогоняет результат через небольшую вспомогательную модель с промптом на естественном языке, чтобы извлечь нужную информацию.

## Когда использовать

- Чтение публичной страницы документации, записи в блоге или RFC, упомянутой в разговоре.
- Извлечение конкретного факта, фрагмента кода или таблицы с известного URL без загрузки всей страницы в контекст.
- Резюмирование release-ноут или чейнджлога из открытого веб-ресурса.
- Проверка публичного API-справочника библиотеки, когда исходников нет в локальном репозитории.
- Переход по ссылке, которую пользователь вставил в чат, чтобы ответить на follow-up вопрос.

## Параметры

- `url` (string, обязательный): полностью сформированный абсолютный URL. Обычный `http://` автоматически апгрейдится до `https://`.
- `prompt` (string, обязательный): инструкция, передаваемая маленькой модели извлечения. Опишите точно, что вытащить со страницы, например «перечисли все экспортируемые функции» или «верни минимальную поддерживаемую версию Node».

## Примеры

### Пример 1: извлечение значения конфигурации по умолчанию

```
WebFetch(
  url="https://vitejs.dev/config/server-options.html",
  prompt="What is the default value of server.port and can it be a string?"
)
```

Инструмент забирает страницу документации Vite, конвертирует в Markdown и возвращает короткий ответ, например «По умолчанию `5173`; принимает только число».

### Пример 2: резюмирование раздела чейнджлога

```
WebFetch(
  url="https://nodejs.org/en/blog/release/v20.11.0",
  prompt="List the security fixes included in this release as bullet points."
)
```

Полезно, когда пользователь спрашивает «что изменилось в Node 20.11», а страница релиза длинная.

## Примечания

- `WebFetch` падает на любом URL, требующем аутентификации, cookies или VPN. Для Google Docs, Confluence, Jira, приватных ресурсов GitHub или внутренних вики используйте специализированный MCP-сервер, предоставляющий аутентифицированный доступ.
- Для всего, что хостится на GitHub (PR, issues, блобы файлов, ответы API), предпочитайте CLI `gh` через `Bash`, а не скрейпинг веб-UI. `gh pr view`, `gh issue view` и `gh api` возвращают структурированные данные и работают с приватными репозиториями.
- Результаты могут быть обобщены, когда забранная страница очень большая. Если нужен точный текст, сузьте `prompt` запросом дословной выдержки.
- На каждый URL применяется самоочищающийся 15-минутный кэш. Повторные вызовы одной и той же страницы в рамках сессии почти мгновенны, но могут вернуть слегка устаревшее содержимое. Если свежесть важна, упомяните это в промпте или переждите кэш.
- Если целевой хост выдаёт кросс-хост редирект, инструмент возвращает новый URL в специальном блоке ответа и не следует за ним автоматически. Перевызовите `WebFetch` с целью редиректа, если всё ещё хотите содержимое.
- Промпт выполняется моделью меньше и быстрее основного ассистента. Держите его узким и конкретным; сложные многошаговые рассуждения лучше обработать, прочитав сырой Markdown самостоятельно после забора.
- Никогда не передавайте секреты, токены или идентификаторы сессий, встроенные в URL, — содержимое страниц и строки запросов, отражённые в выводе, могут логироваться upstream-сервисами.
